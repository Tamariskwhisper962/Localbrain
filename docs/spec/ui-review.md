# UI가 들어갈 여지 검토

## 결론 먼저
UI를 **운영 UI**와 **인사이트 UI** 둘로 나눠 보면 답이 명확하다.
- **운영 UI**(소스 추가·수동 인덱싱·진행상황·검색 테스트·모델 교체) — **데이터가 없어도 첫날부터 유용** → **초기 범위에 포함**.
- **인사이트 UI**(클러스터·FAQ·지식공백) — 질의가 쌓여야 의미 → **나중에 같은 대시보드에 탭 추가**.

권고: **가벼운 로컬 웹 대시보드(운영 탭)부터** 만들고, 인사이트 탭은 데이터가 모인 뒤 붙인다.
(엔진 자체는 여전히 MCP 서버 + CLI로도 구동 가능 — UI는 그 위의 어댑터.)

## UI가 가치 있는 5개 지점

| 지점 | 왜 UI가 좋은가 | 가치 |
|------|---------------|:---:|
| **인사이트 대시보드** | 클러스터·FAQ·인기주제·지식공백은 **본질적으로 시각적**(버블/막대/추이). 자가개선 루프를 눈에 보이게 만든다 | ★★★ |
| **소스 관리** | 폴더 **드래그앤드롭** 추가, 인덱싱 진행률, 무엇이 색인됐는지, 재색인 버튼 | ★★ |
| **검색 플레이그라운드** | 검색 품질 수동 테스트, 청크·점수 확인, k/청크크기 튜닝 (튜닝은 실제 노동 구간) | ★★ |
| **설정** | 임베딩 모델·청크 파라미터·경로 GUI 설정 | ★ |
| **관측(헬스)** | 인덱스 규모·로그·에러 모니터 | ★ |

→ **인사이트**는 시각화 ROI 1위지만 데이터 필요. **운영 4종(소스·인덱싱·검색·설정)은 즉시 유용** → 먼저 만든다.

## 운영 UI 패널 스펙 (초기 범위)

단일 페이지 + 탭. 백엔드(FastAPI)가 `core/` 함수를 JSON으로 노출, 진행상황은 SSE로 스트리밍.

| 탭 | 구성 | 핵심 인터랙션 |
|----|------|--------------|
| **Sources** | 인덱싱된 경로 목록(청크 수·최근 갱신·stale 표시) | 폴더/파일 **경로 추가**(드래그앤드롭 또는 입력), 제거 |
| **Indexing** | "변경분 인덱싱 / 전체 재구축" 버튼, **진행률·ETA·로그 스트림** | **수동 인덱싱 실행**, 취소 |
| **Search (Test)** | 질의 입력 → 결과(점수·출처·스니펫), `k`·필터 조절 | **검색 품질 테스트**, 청크 들여다보기 |
| **Model** | 활성 모델·차원 표시, 모델 드롭다운 | **모델 교체**(전체 재임베딩 경고 + 진행률) — architecture §9 |
| (later) **Insights** | 클러스터·FAQ·인기주제·지식공백 | 데이터 누적 후 추가 |
| (later) **Health** | 인덱스 규모·에러·로그 | 모니터 |

> 요청하신 "소스 추가 · 수동 인덱싱 · 진행상황 · 테스트"가 Sources/Indexing/Search/Model 4탭에 모두 들어간다.

## 웹 UI 동작 방식 (로컬 앱)

> 인터넷 사이트가 아니라 **`127.0.0.1`에서 도는 로컬 앱**. 파일을 읽는 건 브라우저가 아니라 **같은 PC의 백엔드**다.

```
[브라우저 UI]  ──HTTP/SSE──▶  [로컬 백엔드 FastAPI @127.0.0.1]  ──▶ 디스크 파일 직접 읽기
 (경로 지정·버튼·진행표시)         (full 파일시스템 권한, core/ 호출)   ──▶ 벡터 DB(로컬 파일)
```

- 브라우저 = **조종판**(경로 지정·실행·진행표시), 백엔드 = **파일 제자리 읽기 + core 호출 + 벡터DB**.
- 파일은 브라우저로 업로드되지 않는다. 백엔드가 디스크에서 직접 처리.

### 경로 지정 — 브라우저 샌드박스 우회
브라우저는 임의 로컬 경로를 못 읽지만, **백엔드(=내 PC)가 읽으므로 UI는 "경로"만 전달**하면 된다.

| 방법 | 동작 | 평가 |
|------|------|------|
| A. 절대경로 입력 | `C:\...\docs` 타이핑 → 백엔드 검증·인덱싱 | 단순·확실 |
| **B. 서버측 폴더 브라우저** ⭐ | `GET /api/fs/list?path=` 로 디렉터리 목록 → 폴더 트리 렌더 | 백엔드가 내 PC라 **내 실제 폴더 탐색** |
| C. File System Access API | 브라우저가 핸들 획득 후 내용 업로드 | "제자리 인덱싱" 이점 상실, 비권장 |

→ **B + A(폴백)** 권장.

### 흐름 (폴더 추가 → 인덱싱 → 진행)
```
① http://127.0.0.1:8765 접속 (앱 실행 시 자동 오픈)
② Sources: GET /api/fs/list 로 폴더 선택 → POST /api/sources {path,globs,recursive}
③ Indexing: POST /api/index/run → 백엔드 백그라운드 작업(walk→청킹→임베딩→저장)
④ 진행: GET /api/index/stream (SSE) → "12/340 파일·1,520 청크·ETA 00:45" 실시간
⑤ Search: POST /api/search {query,k} → 점수·출처·스니펫
```

### 엔드포인트 ↔ 탭
| 탭 | 엔드포인트 |
|----|-----------|
| Sources | `GET /api/fs/list`, `GET/POST/DELETE /api/sources` |
| Indexing | `POST /api/index/run`, `/cancel`, `GET /api/index/stream`(SSE) |
| Search | `POST /api/search` |
| Model | `GET /api/model`, `POST /api/model/switch`(SSE 진행) |

### 동시성 & 보안
- MCP 서버(stdio)와 웹 백엔드(FastAPI)는 **같은 `core/`+벡터DB를 공유하는 두 어댑터**.
- 쓰기(인덱싱)는 **한 곳(웹 백엔드)으로 모으고** MCP는 읽기(검색) 위주 → 쓰기 충돌 회피. (SQLite WAL로 완화)
- `127.0.0.1` 루프백만 바인딩(외부 비노출), 필요 시 로컬 토큰.

### 대안 — 데스크톱 셸
Tauri/Electron/.NET WebView면 **OS 네이티브 폴더 다이얼로그**를 바로 호출(서버측 폴더 브라우저 불필요). 단 빌드·패키징 비용↑. 가볍게 시작은 localhost 웹로 충분, 배포 욕심 커지면 승격(Phase 3).

## 형태별 옵션 & 작업량

| 형태 | 설명 | 작업량 | 적합 |
|------|------|--------|------|
| **CLI only** | UI 없음, 파워유저용 | (산정에 포함) | MVP |
| **로컬 웹 대시보드** ⭐ | FastAPI가 JSON 제공 + 단일 페이지 프론트(localhost) | **3~5일** | 인사이트 성숙 후 |
| 데스크톱 앱 | Tauri/Electron 또는 Atlas식 .NET WebView | 1~2주 | 광범위 배포 시 |
| TUI (textual) | 터미널 UI | 2~3일 | 니치 |

> 웹 대시보드는 **plain HTML+htmx면 더 적게, 풀 React면 더 많이**. 개인용이면 가벼운 쪽 권장.

## 설계상 전제 — 코어/어댑터 분리
[`architecture.md`](architecture.md) §2처럼 핵심 로직을 라이브러리로 두면,
웹 대시보드는 **core 함수를 호출하는 얇은 어댑터**가 된다 → UI 추가 비용이 작다.
MCP·CLI·UI가 같은 `core/`를 공유.

## 디자인 연속성
대시보드 UI는 이미 만든 **`localbrain/index.html`의 모던 다크 테마**(CSS 변수·카드·표 스타일)를
그대로 재사용하면 톤이 일관되고 초기 비용도 절감된다.

## 권고 로드맵
1. **Phase 1**: core 엔진 + MCP + CLI + **운영 웹 UI(Sources·Indexing·Search·Model 4탭)**. — 운영 UI는 즉시 유용
2. **Phase 2**: 질의가 쌓이면 같은 대시보드에 **Insights 탭** 추가
3. **Phase 3(선택)**: 배포 욕심이 생기면 데스크톱 셸(Tauri/.NET WebView)로 승격

## 작업량 영향
- 운영 웹 UI(4탭 + SSE 진행 스트리밍, plain HTML/htmx 또는 가벼운 SPA): **+4~6일**
  (백엔드는 core 재사용, 진행 스트리밍이 약간의 추가 비용)
- 따라서 **MVP + 간단 운영 UI**는 헤드리스 MVP(~1주) 대비 **약 1.5~2주** 규모.

> 콜드스타트 주의는 **인사이트 탭에만** 해당(데이터 없으면 빈 화면). **운영 탭은 데이터와 무관하게 첫날부터 동작**하므로 먼저 만드는 게 맞다.
