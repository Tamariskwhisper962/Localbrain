01 Ollama / 로컬 LLM

대규모 언어 모델을 로컬에서 쉽게 실행하게 해주는 오픈소스 도구. "LLM계의 Docker"라고 보면 된다.

핵심 특징

  • 로컬 실행 — 인터넷 없이 내 PC에서 구동, 프라이버시 보장 + API 비용 0
  • 간편함 — ollama run llama3 한 줄로 다운로드부터 실행까지
  • 다양한 모델 — Llama, Mistral, Gemma, Qwen, Phi, DeepSeek 등
  • 크로스 플랫폼 · GPU 가속 — Windows/macOS/Linux, NVIDIA·AMD·Apple Silicon 지원

기본 명령어

ollama run llama3      # 실행 (없으면 자동 다운로드)
ollama pull mistral    # 다운로드만
ollama list            # 설치된 모델 목록
ollama ps              # 실행 중인 모델

API 서버로 활용

설치 시 http://localhost:11434에 REST API 서버가 자동으로 뜬다. OpenAI 호환 API(/v1/chat/completions)도 제공해, 기존 OpenAI SDK 코드의 엔드포인트만 바꿔 로컬 모델로 전환할 수 있다.

curl http://localhost:11434/api/generate -d '{
  "model": "llama3",
  "prompt": "안녕하세요"
}'

Modelfile — 커스텀 모델

Dockerfile처럼 시스템 프롬프트·파라미터를 정의해 나만의 모델을 만든다.

FROM llama3
PARAMETER temperature 0.7
SYSTEM "너는 친절한 한국어 비서야."

# 생성 & 실행
# ollama create my-assistant -f Modelfile
# ollama run my-assistant
✔

언제 좋은가? 민감 데이터를 외부로 안 보내고 처리할 때, API 비용 없이 실험할 때, 오프라인 환경, 그리고 — 바로 다음 주제인 로컬 RAG의 엔진으로.

02 로컬 RAG의 원리

RAG(Retrieval-Augmented Generation) = LLM이 답하기 전에 내 문서에서 관련 정보를 검색해 그 근거로 답하게 하는 기법. 모든 과정을 내 PC에서 하면 "로컬 RAG".

왜 필요한가

  • 지식 한계 — 학습 이후 정보나 비공개 사내 문서는 모델이 모른다
  • 환각(Hallucination) — 모르는 걸 그럴듯하게 지어낸다

RAG는 "관련 부분을 찾아 → 그걸 근거로 답변"하게 해 이 둘을 해결한다. 재학습(파인튜닝) 없이 최신·전용 지식을 주입할 수 있는 게 큰 장점이다.

2단계 흐름

① 인덱싱 (사전 준비)
   문서(PDF·TXT·MD)
      │  청킹(Chunking) — 작은 조각으로 분할
      ▼
   텍스트 청크들
      │  임베딩(Embedding) — 각 청크를 벡터로 변환
      ▼
   벡터들 ───────────────▶ [ 벡터 DB 에 저장 ]

② 질의 (실제 사용)
   사용자 질문
      │  질문도 똑같이 임베딩
      ▼
   질문 벡터  ──유사도 비교(코사인)──▶ [ 벡터 DB ]
                                         │ 가까운 청크 검색
                                         ▼
   관련 청크 + 원래 질문 ──▶ [ LLM ] ──▶ 근거 기반 답변

"유사도 비교"의 정체 = 벡터(임베딩)

청크를 쪼개기만 해선 비교가 안 된다. 각 청크를 임베딩 모델로 벡터(숫자 배열)로 바꾸면, 의미가 비슷할수록 벡터 공간에서 가까이 위치한다. 그래서 키워드가 정확히 일치하지 않아도 의미 기반 검색이 된다.

"강아지는 귀엽다"  →  [0.21, -0.88, 0.45, ...]  ┐ 가까움
"개는 사랑스럽다"  →  [0.19, -0.85, 0.48, ...]  ┘ (의미 유사)
"오늘 환율은..."   →  [0.91,  0.12, -0.70, ...]   멀리 떨어짐
ℹ

검색의 주체는 LLM이 아니다. 검색은 벡터 DB + retriever(검색 로직)가 하고, LLM은 찾아온 청크를 읽고 답변을 생성할 뿐이다. (단, 요즘은 LLM이 "검색이 필요한지 판단 → 도구 호출"까지 하는 에이전틱 RAG로 경계가 흐려지고 있다.)

한 문장 요약 — 내 문서를 청크로 쪼개고 → 각 청크를 벡터로 변환해 벡터 DB에 인덱싱해두었다가 → 질문이 들어오면 질문도 벡터로 바꿔 의미가 가까운 청크를 검색해 LLM에게 근거로 건네주는 구조.

03 RAG 구축 도구

거창한 유료 도구가 필요한 게 아니라, 대부분 무료 오픈소스 조합이다. PC 한 대 + Python이면 시작할 수 있다.

4가지 구성 요소

구성 요소역할로컬 예시 (무료)
① 임베딩 모델텍스트 → 벡터 변환Ollama nomic-embed-text, bge-m3
② 벡터 DB벡터 저장 + 유사도 검색Chroma, FAISS, Qdrant, LanceDB
③ LLM검색 내용으로 답변 생성Ollama llama3, qwen
④ 오케스트레이션위 셋을 연결하는 흐름 제어LangChain, LlamaIndex (또는 직접 코딩)

①과 ③은 Ollama 하나로 둘 다 해결되고, ②는 pip install로 설치한다.

난이도별 선택지

방식특징대표
라이브러리 조합직접 코딩, 모든 단계 세밀 제어LangChain/LlamaIndex + Ollama + Chroma
올인원 도구설치만 하면 GUI로 문서 끌어다 놓기AnythingLLM, Open WebUI, GPT4All, LM Studio
클라우드 관리형운영 부담↓ 그러나 데이터 외부 유출Pinecone, Weaviate Cloud, Dify, Flowise

최소 Python 예시 (LangChain + Ollama + Chroma)

# pip install langchain langchain-community langchain-ollama chromadb
from langchain_community.document_loaders import TextLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_ollama import OllamaEmbeddings, ChatOllama
from langchain_community.vectorstores import Chroma
from langchain.chains import RetrievalQA

# 1. 문서 로드 & 청킹
docs = TextLoader("my_document.txt", encoding="utf-8").load()
chunks = RecursiveCharacterTextSplitter(
    chunk_size=500, chunk_overlap=50).split_documents(docs)

# 2. 임베딩 & 벡터 DB 저장 (모두 로컬)
embeddings = OllamaEmbeddings(model="nomic-embed-text")
vectorstore = Chroma.from_documents(chunks, embeddings)

# 3. RAG 체인
llm = ChatOllama(model="llama3")
qa = RetrievalQA.from_chain_type(
    llm=llm, retriever=vectorstore.as_retriever(search_kwargs={"k": 3}))

# 4. 질문
print(qa.invoke({"query": "이 문서의 핵심 내용은?"})["result"])
ℹ

먼저 ollama pull nomic-embed-text 와 ollama pull llama3 로 모델을 받아둔다.

품질을 높이는 포인트

  • 청킹 전략 — 너무 크면 잡음, 너무 작으면 맥락 끊김 (보통 300~800 토큰 + 오버랩)
  • 좋은 임베딩 모델 — 한국어라면 다국어에 강한 bge-m3 권장
  • 검색 개수(k) 조정 · 재순위화(Reranking) · 하이브리드 검색(벡터+키워드)

코딩 없이 체험 AnythingLLM + Ollama   만들며 배우기 LangChain + Chroma   가장 단순 LlamaIndex + FAISS

04 RAG vs MCP

둘 다 "LLM의 한계를 외부 자원으로 메운다"는 점이 비슷하게 들리지만 — RAG는 기법(방법론), MCP는 표준 규격(프로토콜)으로 애초에 같은 층위가 아니다.

구분RAGMCP (Model Context Protocol)
본질기술/패턴 — 어떻게 답할까프로토콜/표준 — 어떻게 연결할까
목적관련 문서를 찾아 답변 정확도↑LLM이 외부 도구·데이터에 접근
방향주로 읽기(검색)읽기 + 쓰기/실행(행동)
방식임베딩 → 벡터 검색 → 프롬프트 주입클라이언트-서버 표준 메시지 교환
예시사내 문서 Q&A 챗봇Claude가 GitHub·DB·파일시스템 조작
💡

비유 — RAG는 도서관에서 관련 책 페이지를 책상에 펼쳐주는 것(정보 제공). MCP는 LLM에게 전화기와 리모컨을 쥐여주는 것(행동 능력). MCP를 "USB-C 포트"에 비유하기도 한다: 한 번 MCP 서버로 만들면 어떤 LLM 앱이든 표준 단자에 꽂아 쓴다.

경쟁이 아니라 보완 — MCP가 RAG를 감쌀 수 있다

[ LLM 앱 / Claude ]
       │  MCP 프로토콜로 호출 ("검색해줘")
       ▼
[ MCP 서버: "문서 검색" ]   ← 이 서버 내부가 사실 RAG 시스템
       ├─ 벡터 DB 검색
       └─ 관련 청크 반환

즉 RAG는 서버 내부에서 실제로 검색하는 로직이고, MCP는 그 기능을 LLM 앱에 연결·호출하는 통로다. RAG는 "무엇을 알려줄까(데이터)", MCP는 "어떻게 연결·행동시킬까(인터페이스)"에 초점이 있다.

05 자가 개선 RAG 루프

대화에서 나온 질문·피드백을 RAG에 다시 반영해 "살아있는" 시스템으로 진화시킬 수 있다. 실무 챗봇·고객지원 봇에서 이미 널리 쓰이는 표준 패턴이다.

성격이 다른 두 가지 목표

목표무엇을 반영?성격
A. 지식 보강검증된 Q&A를 새 문서로 재인덱싱검색 대상(지식)을 늘림
B. 운영 최적화"어떤 질문/문서가 자주 쓰이나" 통계시스템 개선용 메타데이터

A. 지식 축적 — 검증된 Q&A 재인덱싱

if user_feedback == "good":
    qa_text = f"질문: {question}\n답변: {answer}"
    doc = Document(page_content=qa_text,
                   metadata={"source": "verified_qa", "date": today})
    vectorstore.add_documents([doc])   # 벡터 DB에 추가
⚠

오염(feedback poisoning) 주의. 검수 없이 LLM 답변을 무조건 다시 넣으면 환각이 지식으로 굳어진다. 반드시 "사람 승인" 또는 "👍 받은 것만" 반영할 것.

B. 사용 패턴 분석 — 검색 로그 활용

분석활용
질문 벡터 군집화(클러스터링)자주 묻는 유사 질문 그룹 발견 → FAQ 자동 생성
검색 빈도 집계자주 요청되는 문서/청크 파악 → 캐싱·상위 노출
검색됐는데 답 못 한 질문지식 공백(missing knowledge) 발견 → 문서 보충
👎 받은 답변청킹/검색 품질 개선 지점 파악
ℹ

"자주 묻는 질문 그룹화"는 새 질문을 기존 질문들과 벡터 유사도로 비교해 묶는 것 — RAG의 임베딩 기술을 그대로 재활용한다.

둘을 합치면 = 자가 개선 루프 (Adaptive / Self-improving RAG)

   사용자 대화 ──▶ RAG 답변 ──▶ 피드백/로그 수집
        ▲                            │
        │              ① 자주 묻는 질문 군집화
        │              ② 검증된 Q&A를 지식에 추가
        │              ③ 부족한 문서 보충
        │                            │
        └──────  개선된 RAG로 순환  ◀──┘

구현은 특별할 게 없다. 기존 RAG에 ① 로그 저장소(SQLite/PostgreSQL)와 ② 반영 파이프라인(예: 매일 밤 로그 분석 → 승인 Q&A 추가 · FAQ 갱신)을 붙이는 정도.

전문 도구: Langfuse · LangSmith · RAGAS(평가) — RAG 품질 모니터링·로그 분석에 특화. 이런 도구가 존재한다는 것 자체가 이 패턴이 표준임을 보여준다.

06 MCP로 RAG 구현 · 임베딩의 핵심

Claude Code 같은 클라이언트에 MCP로 RAG를 붙일 때 실제 구조는 어떻게 되며, 무엇이 성능을 좌우하는가.

역할 분리 — 임베딩 모델 ≠ 생성 LLM

종류하는 일예시
임베딩 모델 (인코더)텍스트 → 벡터로 변환만nomic-embed-text, bge-m3
생성 LLM텍스트를 생성/추론Llama3, Qwen, Claude

임베딩은 생성 LLM이 하는 게 아니라 임베딩 전용 모델이 한다. 그리고 임베딩 모델은 "분석"하는 게 아니라 의미를 숫자 배열로 인코딩할 뿐이다.

⚠

흔한 오해 — "LLM이 유사도를 비교해 찾아낸다"? 아니다. 가장 가까운 벡터를 찾는 일은 벡터 DB의 순수 수학 연산(코사인 거리 계산)이며, 지능이 개입하지 않는 결정론적 작업이다. 같은 입력이면 항상 같은 결과.

Claude Code + MCP 실제 구조

생성·추론 두뇌가 이미 Claude이므로, MCP 서버 안에 생성 LLM을 올릴 필요가 없는 경우가 많다. 서버는 "임베딩 모델 + 벡터 DB"면 충분하다.

[ Claude Code ]  ← 생성·추론 담당 (LLM 역할 전부)
      │  MCP로 "search(질문)" 호출
      ▼
[ MCP 서버 ]
      ├─ 임베딩 모델 : 질문을 벡터로 변환   (작은 인코더)
      ├─ 벡터 DB     : 내 자료 벡터와 거리 비교 → 가까운 청크 검색 (수학)
      └─ 청크 텍스트 반환
      │
      ▼
[ Claude ] 가 받은 청크를 근거로 최종 답변 생성
✔

이 구조의 장점: 무거운 생성 LLM을 로컬에 안 올리고, 가볍고 빠른 임베딩 모델만 로컬에 둔 채 Claude의 강력한 추론을 그대로 쓸 수 있다. (질문 임베딩도 클라이언트가 아니라 서버 안에서 일어난다.)

그래서 가치의 핵심 = 임베딩 모델의 성능

RAG는 "검색 → 생성"이 직렬로 연결된 파이프라인이라, 최종 품질은 두 단계 중 약한 쪽에 묶인다.

[임베딩/검색]  →  [생성(Claude)]
   품질 C            품질 A
        └─── 최종 답변 품질 ≈ min(C, A) ───┘

Claude가 아무리 똑똑해도 검색이 엉뚱한 청크를 가져오면 잘못된 근거로 답할 수밖에 없다(garbage in, garbage out). 그래서 임베딩 모델이 사실상 RAG 성능의 상한선을 정한다.

임베딩 모델 선택 기준

기준설명
언어 지원한국어 문서라면 다국어에 강한 모델 필수 (bge-m3, multilingual-e5)
MTEB 점수임베딩 검색 성능 벤치마크. Retrieval 부문 점수가 핵심 지표
차원 수 / 속도차원↑ 정확도↑ 그러나 저장·검색 비용↑. 로컬이면 속도·메모리도 고려
컨텍스트 길이한 번에 임베딩할 수 있는 청크 최대 길이
도메인 적합성법률·의료·코드 등은 해당 분야 특화 모델이 유리

한국어 기본값 bge-m3   다국어 multilingual-e5-large   가볍고 빠름 nomic-embed-text

ℹ

재인덱싱 주의 — 임베딩 모델을 바꾸면 기존 벡터를 전부 다시 임베딩해야 한다(다른 모델의 벡터는 호환 안 됨). 처음부터 한국어에 강한 모델로 시작하면 나중에 갈아엎는 수고를 줄인다.

한 줄 정리 — 임베딩 모델은 "무엇을 찾아올지 결정하는 눈", Claude는 "찾아온 걸 해석하는 두뇌". 눈이 흐리면 두뇌가 좋아도 소용없으니, RAG의 핵심은 임베딩 모델의 품질·데이터·검색 설계다.

07 인덱싱형 vs 페더레이션형 (M365 사례)

RAG의 "검색"은 꼭 미리 인덱싱해둘 필요가 없다. 질문할 때마다 원본을 실시간으로 찾아오는 페더레이션 방식도 있고, Microsoft 365가 둘을 모두 쓰는 좋은 사례다.

두 가지 검색 패턴

[A] 인덱싱형 (미리 임베딩 / M365 Semantic Index)
    문서 → 미리 임베딩 → 벡터·의미 인덱스 저장 → 질문 시 검색
    + 빠름, 대규모     − 동기화 필요, 최신성 지연

[B] 페더레이션형 (실시간 / M365 federated 커넥터)
    질문 발생 → 원본 시스템에 실시간 질의 → 그때그때 받아옴
    + 항상 최신, 데이터를 원본에 그대로 둠(거버넌스 유지)
    − 매 질의마다 외부 호출, 원본 검색 성능에 의존

Synced vs Federated 커넥터

Synced (동기화형)Federated (페더레이션형)
데이터 위치중앙(MS Graph)에 복사·저장원본 시스템에 그대로 둠
인덱싱미리 인덱싱안 함
검색 시점저장된 인덱스에서질문할 때 실시간 질의
최신성동기화 주기만큼 지연항상 최신
ℹ

도서관 비유 — Synced는 모든 책을 미리 복사해 자기 서가에 꽂아두고 검색. Federated는 책을 안 가져오고 목록만 알다가, 물어보면 그제서야 원본 소장처에 전화해 받아온다(항상 최신, 매번 요청).

Microsoft 365는 어떻게 동작하나 — 2개 층으로 보기

[ 층 A ] Claude Code ──실시간 MCP 호출──▶ M365 (MCP 서버)
[ 층 B ]                                M365 내부에서 실제로 어떻게 찾나?
  • 층 A (Claude Code ↔ M365) — Claude Code는 MCP 클라이언트일 뿐, M365를 자기 쪽에 인덱싱하지 않는다. 항상 실시간 호출이며, 인덱스를 가진 쪽은 M365다.
  • 층 B (M365 내부) — 여기서 인덱싱/페더레이션이 갈린다.
M365가 다루는 대상방식
M365 자체 문서 (SharePoint·OneDrive·Teams·메일)미리 인덱싱됨 (Semantic Index)
외부 데이터 — federated 커넥터실시간 페더레이션 (인덱싱 X)
외부 데이터 — synced 커넥터Graph에 동기화·인덱싱 후 활용
⚠

"인덱스를 쓰느냐"는 어떤 API를 부르냐에 달림. MCP 서버가 Copilot Retrieval API / Microsoft Search API를 부르면 Semantic Index 기반 의미 검색이 되지만, 원시(raw) Graph API(파일 목록·메일 가져오기)만 부르면 인덱스 미사용 — "검색"이 아니라 단순 "fetch"다. 같은 "M365 MCP"라도 공식 서버냐 커뮤니티 래퍼냐에 따라 다르다.

한 줄 정리 — MCP는 "검색 방식을 규정하지 않는 통로"라, 그 너머가 미리 만든 인덱스든 실시간 페더레이션이든 상관없다. M365는 자체 문서는 인덱싱형, 외부 federated 커넥터는 페더레이션형으로 동시에 운영한다.

09 DB 데이터에 RAG 적용하기

"DB 전체를 임베딩 인덱싱"은 대개 정답이 아니다. 먼저 "이 데이터에 임베딩이 맞는 방식인가"부터 따져야 한다.

정형 vs 비정형 — 검색 방식이 갈린다

데이터 성격예시적합한 방식
정형(structured)매출액·날짜·카테고리·ID·수량SQL / Text-to-SQL (임베딩 불필요)
비정형 텍스트상품 설명·고객 문의·리뷰·메모임베딩 RAG (의미 검색)
ℹ

임베딩은 자연어 텍스트에 강하고, 숫자·날짜·코드 같은 정형 값엔 부적합하다. "지난달 매출 top 10"은 임베딩이 아니라 SQL 한 줄이 정확·빠르고, "환불 요청이 잦은 불만 유형"은 문의 본문의 의미 검색이 필요하다.

전체 말고 — 선택적 / 혼합

① 텍스트 컬럼만 선택적 임베딩 — 테이블 전체가 아니라 자연어 컬럼만 임베딩하고, 정형 컬럼은 메타데이터(필터)로.

[ 주문 테이블 ]
  order_id, date, amount, category  ← 정형 → 필터 / WHERE 조건
  customer_review (텍스트)          ← 비정형 → 임베딩

② 하이브리드: SQL 필터 + 의미 검색

"작년 4분기(=정형 필터) 주문 중, '배송 지연' 불만(=의미 검색)"
   → WHERE date BETWEEN ... (SQL)  +  리뷰 텍스트 벡터 검색

정형 조건으로 후보를 좁힌 뒤 그 안에서 의미 검색을 돌리면 정확도·속도가 함께 올라간다.

인덱싱한다면 — "한 번에 전부" 말고 증분

  • 신규/변경 행만 증분 인덱싱 (타임스탬프·CDC·트리거 활용)
  • 행(row) 하나 = 문서(청크) 하나로 보고, 관련 필드를 합쳐 텍스트화
  • DB가 자주 바뀌면 전체 재임베딩은 비용·시간이 큼

또는 — 인덱싱하지 않는 선택지 (페더레이션)

M365처럼 미리 인덱싱하지 않고 질문 시점에 실시간 질의하는 방법. 정형 데이터엔 오히려 이쪽이 적합한 경우가 많다.

[ Text-to-SQL MCP 서버 ]
  질문 → LLM이 SQL 생성 → DB 실시간 실행 → 결과 반환
  + 항상 최신, 정확한 집계/계산
  − 복잡한 자연어 질의는 SQL 생성 정확도에 의존

결정 가이드

상황추천
정형 데이터 (집계·필터·정확 조회)Text-to-SQL (인덱싱 X)
비정형 텍스트 (의미·유사 검색)해당 텍스트 컬럼만 임베딩
둘이 섞임 (대부분의 실무)하이브리드 (SQL 필터 + 의미 검색)
데이터가 자주 바뀜증분 인덱싱 또는 실시간 페더레이션

한 줄 정리 — "DB 전체 임베딩"은 대개 비효율. 정형은 SQL/Text-to-SQL로, 비정형 텍스트 컬럼만 선택적으로 임베딩하고, 변화가 잦으면 증분 인덱싱이나 실시간 페더레이션을 쓴다. 전부 인덱싱이 아니라 무엇을 어떤 방식으로 검색할지 나누는 설계가 핵심.
